Especificación Técnica: Plataforma Automatizada "GNOME Beta Store"Este documento describe la arquitectura, el flujo de trabajo y los requisitos de desarrollo para crear una tienda de extensiones de GNOME alternativa, automatizada, segura y de coste cero utilizando la infraestructura gratuita de GitHub (Actions, Issues y Pages) y APIs externas.1. Arquitectura General (Dos Repositorios)Para garantizar la seguridad, escalabilidad y limpieza del proyecto, el sistema se divide en dos repositorios independientes:Repositorio A: gnome-beta-store (Backend, Base de Datos y Orquestación)Función: Actúa como el motor del sistema. Almacena los archivos .zip de las extensiones validadas, sus recursos gráficos (iconos y capturas), mantiene un índice centralizado en formato JSON (extensions.json) y aloja las GitHub Actions que realizan el análisis de seguridad.Base de datos estática: Un archivo extensions.json en la raíz del repositorio que guarda la información de las extensiones activas (UUID, versión, nombre, descripción, enlace de GitHub, enlace promocional opcional, ruta del icono, array de capturas de pantalla y URL de descarga del ZIP en raw.githubusercontent.com).Repositorio B: gnome-beta-web (Frontend y Catálogo)Función: Interfaz web estática para el usuario final, alojada gratuitamente en GitHub Pages.Comportamiento: Realiza una petición fetch en tiempo real al archivo extensions.json del Repositorio A para renderizar las extensiones disponibles con sus imágenes, enlaces a repositorios, webs publicitarias y genera comandos dinámicos de instalación para la terminal.2. Flujo de Subida del Desarrollador (GitHub Issues)Para evitar que los usuarios tengan que clonar el repositorio, la subida de nuevas extensiones o actualizaciones se realiza exclusivamente mediante GitHub Issues:El desarrollador abre un Issue en el Repositorio A utilizando una plantilla de formulario estructurada (Issue Form).El formulario solicita obligatoriamente los siguientes campos:UUID de la extensión: (ej. mi-extension@autor.com).Nombre Claro: Nombre comercial o público de la extensión.Descripción: Breve resumen de qué hace la extensión para mostrar en la web.Enlace de GitHub: URL del repositorio de código de la extensión (obligatorio para transparencia de cara al usuario).Enlace Promocional o Página Web: URL opcional para marketing, donaciones o documentación propia.Archivo ZIP: El paquete comprimido con el código fuente de la extensión.Icono (icon.png): Imagen cuadrada para representar la extensión en el catálogo.Demos / Capturas: Una o más capturas de pantalla (demo1.png, demo2.png, etc.) que muestren la interfaz o funcionamiento de la extensión.La creación del Issue con una etiqueta específica (ej. nueva-extension) activa automáticamente una GitHub Action.3. Lógica del Pipeline de Validación (GitHub Actions)La GitHub Action se ejecuta en un entorno virtual gratuito de GitHub y debe realizar los siguientes pasos de forma secuencial. Si alguno falla, el proceso se detiene y se notifica al usuario.Paso 1: Descarga, Extracción y Organización de AssetsEl script lee el cuerpo del Issue, extrae las URLs de las respuestas del formulario, el archivo .zip, la imagen icon.png y las capturas adjuntadas (demo*.png).Descarga y organiza temporalmente los archivos:El .zip se descarga para su análisis.icon.png se renombra a [UUID].png y se prepara para la carpeta assets/icons/.Las imágenes de demo se renombran ordenadamente a demo1.png, demo2.png... y se preparan para guardarse en assets/demos/[UUID]/.Descomprime el archivo .zip de forma temporal.Lectura de Metadatos: El script debe leer obligatoriamente el archivo metadata.json extraído del .zip para identificar las versiones compatibles de GNOME Shell declaradas en el array "shell-version" (ej. ["45", "46", "47", "48", "49"]).Paso 2: Pre-filtrado Estricto (Optimización de Tokens de IA)Para evitar saturar o agotar los límites gratuitos de la API de NVIDIA con extensiones pesadas, el script de validación local debe aplicar las siguientes reglas antes de llamar a la IA:Filtro de extensiones de archivo: Solo se leerá el contenido de archivos de código fuente de texto plano (.js, .ts, .json). Se ignoran por completo imágenes, sonidos, traducciones (.po, .mo) o archivos binarios.Exclusión de dependencias de terceros: Se omitirá la lectura de cualquier archivo ubicado dentro de carpetas de dependencias comunes (como node_modules/, vendor/ o subcarpetas de librerías externas) y archivos minificados (*.min.js). Solo se audita el código de lógica propia escrito por el autor.Cortafuegos de tamaño máximo: Si tras aplicar los filtros anteriores, el tamaño total del texto de código acumulado supera un límite prudencial (ej. 50,000 caracteres), el script abortará la ejecución automáticamente para proteger la API de NVIDIA. El Issue se cerrará con un aviso indicando que el tamaño excede la auditoría automática.Paso 3: Auditoría con IA integrada mediante Fetch Dinámico Condicional de GJSPara que la IA de NVIDIA evalúe con precisión si el código de la extensión es moderno, seguro y compatible con las API actuales de GNOME, el pipeline implementará un sistema de consulta condicional:Consulta del Índice GJS: El script audit.js descarga el índice de documentación markdown desde:https://mdpedia.inled.es/raw/gjs.guide/_index.mdDescarga de Guías Estáticas (Siempre Obligatorias): Independientemente de la versión de la extensión, el script descarga siempre las guías generales fundamentales de la wiki para inyectarlas en la IA:Estructura estándar y ciclo de vida de una extensión (enable() / disable()).Guía de migración a ESM (módulos de JavaScript modernos requeridos desde GNOME 45).Normas generales de revisión y buenas prácticas de seguridad.Descarga Condicional de Cambios por Versión: El script analiza las versiones de GNOME Shell extraídas del "shell-version" del metadata.json (ej: ["46", "47"]). Tras leer el índice _index.md, busca y descarga de forma dinámica los archivos de cambios relativos a dichas versiones, tales como:Cambios y novedades de la API en GNOME 45, 46, 47, 48, etc., según aplique.Esto dota al LLM de la información exacta sobre las APIs que se añadieron o se declararon obsoletas (deprecated) en las versiones específicas que el desarrollador afirma soportar.Inyección de Contexto (Context Injection): El script concatena la información relevante de estas guías en el prompt del sistema de la API de NVIDIA.Configuración del Prompt de Sistema: Se instruye a la IA utilizando este contexto inyectado. Debe evaluar el código de la extensión contrastándolo directamente con las guías y APIs de GJS descargadas, buscando:Incompatibilidades críticas: Uso de imports obsoletos o estructuración pre-ESM que romperán el shell en las versiones declaradas.Seguridad: Ofuscación de código sospechosa (eval(), Base64 extraños).Privacidad: Accesos indebidos del sistema de archivos o sockets de red no justificados.Formato de Respuesta Exigido: La IA debe responder única y estrictamente en formato JSON con la estructura: {"apta": boolean, "motivo": "explicación de la decisión técnica basada en las guías GJS"}.Temperatura: Configurada a 0.1 para un veredicto puramente técnico y determinista.Paso 4: Análisis de Malware (VirusTotal API)Si la IA de NVIDIA aprueba la extensión, el script toma el archivo .zip completo y lo envía a analizar a la API gratuita de VirusTotal.El script consulta el reporte del análisis y verifica que el contador de positivos (detecciones de virus o exploits) sea exactamente 0.Paso 5: Publicación Automatizada y CierreSi el análisis falla (IA o VirusTotal detectan problemas): El bot añade un comentario detallado en el Issue explicando detalladamente el motivo del rechazo (especificando si fue por reglas de GJS incumplidas o por alertas de seguridad) y cierra el Issue como "no planificado".Si el análisis es exitoso:Mueve el archivo .zip a extensions/.Guarda el icono en assets/icons/[UUID].png y las capturas en assets/demos/[UUID]/.Actualiza extensions.json en la raíz incluyendo los nuevos metadatos recibidos del Issue (nombre, descripción, enlace de GitHub, web publicitaria, icono local, array de capturas locales y el enlace de descarga del ZIP).Realiza un git commit y git push de forma automática al Repositorio A.Comenta en el Issue felicitando al autor por cumplir con los estándares de GJS y publica la extensión de inmediato cerrando el Issue.4. Diseño y Funcionamiento de la Web (Repositorio B)La web estática cargada en GitHub Pages realiza la experiencia del usuario final sin necesidad de base de datos activa:Lectura del catálogo: Al cargar la página, se realiza un fetch dinámico al archivo extensions.json alojado en el Repositorio A.Renderizado de tarjetas enriquecidas: Muestra la ficha de cada extensión con su nombre real, descripción y el icono cargado desde assets/icons/[UUID].png. Al abrir el detalle, se muestra el carrusel de capturas descargadas en assets/demos/[UUID]/.Enlaces e Información: En cada tarjeta o modal de detalle, se incluirán botones bien visibles para acceder directamente al repositorio de GitHub de la extensión, y de forma opcional, un enlace a su sitio promocional/página web.Comando de instalación "Un Clic" (Evitando bloqueos de navegador):Ofrece un botón visual de "Copiar Comando de Instalación".Al hacer clic, copia al portapapeles el siguiente comando optimizado con --force:wget -qO- [URL_DEL_ZIP_EN_REPO_A] > /tmp/ext.zip && gnome-extensions install --force /tmp/ext.zip && gnome-extensions enable [UUID_DE_LA_EXTENSION] && rm /tmp/ext.zipInstalación: El usuario pega el comando en su terminal para completar la instalación de manera silenciosa y limpia.5. Instrucciones de Implementación para el Generador (Gemini CLI)Al programar esta solución, asegúrate de cumplir con las siguientes directrices técnicas:Fetch dinámico inteligente: El script audit.js debe implementar una rutina que parsee recursivamente el _index.md descargado de https://mdpedia.inled.es/raw/gjs.guide/... para mapear las rutas relativas correspondientes a las guías generales y de cambios de las versiones objetivo, realizando la descarga asíncrona de forma ágil y ordenada en el runner.Gestión de Secretos: Las credenciales secrets.NVIDIA_API_KEY y secrets.VT_API_KEY deben inyectarse de forma segura en las variables de entorno de la Action.Manejo de Imágenes en el Commiter: El bot de Git debe realizar el push tanto del archivo .json de la base de datos como de todos los archivos binarios de imágenes (.png) recibidos y validados.
